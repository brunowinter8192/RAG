# Eval Suite Extension — Session Process (2026-05-10)

## Drift Discovery

Opening `decisions/retrieval03_fusion.md` and `decisions/retrieval04_reranking.md` revealed NDCG@3/NDCG@10/Recall@10 tables for two March 2026 datasets (SearXNG 30q/26088 chunks, qwen3_paper 15q/66 chunks). No corresponding code exists in `dev/retrieval/` — `A_retrieval_eval.py` was written in April 2026 and only tracks binary doc_recall and snippet_recall. The March data came from a separate eval setup that was never committed to the repo.

These tables appeared in the Evidenz sections of two decision files — i.e., as live IST evidence — but they are not reproducible from the current codebase. This is a structural drift: decision files contain evidence that cannot be re-validated, contradicting the principle that Evidenz should reflect what the current pipeline has actually measured.

Removal was the correct call. The March data had served its purpose (driving the decision to use CC α=0.8 and keep rerank=False as default); that decision is now codified in the Recommendation sections with no dependency on the March numbers.

## RAG Reference Paper Findings

Four papers from the `RAG_reference` collection were consulted to define the metric extension:

**RAGAS Evaluation Framework:** documents a context_recall + context_precision metric pair. Binary relevance via LLM judge is acceptable for small GT sets. The document-level GT structure (expected_documents list per query) matches the RAGAS pattern. Prescribes retrieval-only eval as the first evaluation stage.

**RAG Evaluation Survey 2025:** taxonomy of retrieval metrics: document-level (MRR, MAP) and chunk-level (Recall@K, NDCG@K). Key finding: NDCG@K is the most discriminative single metric for ranked retrieval. Distinguishes retrieval eval (our scope) from generation eval (out of scope for this session).

**Pipeline Optimization (arxiv:2511.22240):** uses Acc@3 and NDCG@3 on 11,000 synthetic QA pairs. Sets the scale reference: our 20-query manual GT is 550× smaller. Provides the benchmark for chunking comparison (+5pp Acc@3 at 512 vs 2000 chars) and reranker impact (+9.4pp Acc@3). Also shows that synthetic GT at scale is feasible via LLM generation.

**Fusion Functions for Hybrid Retrieval (Bruch et al. 2023):** direct methodological precedent for using NDCG@3 and NDCG@10 as fusion comparison metrics. The April 2026 RAG_MCP sweep adopted this methodology but only reported binary doc_recall and snippet_recall. The metric extension brings the eval suite in line with the paper's measurement standard.

## Methodology Choices

**NDCG@K with binary relevance:** rel=1 if hit.document ∈ expected_documents, else 0. Binary relevance is the pragmatic choice: manual GT is at the document level (expected_documents per query), not at the graded chunk level. Graded relevance (e.g. rel=2 for exact snippet, rel=1 for doc match) is listed as an open question for future iteration.

**IDCG via total_relevant from DB:** the standard IR approach — IDCG is computed using the total number of relevant items in the corpus, not just the number that appeared in the returned list. This makes NDCG comparable across queries with different collection sizes. Required a DB query (`src.rag.db.query_documents`) at the start of each eval run to count chunks per expected_document.

**Recall@K as chunk-level metric (NOT binary doc_recall):** the existing `doc_recall` is a binary per-query hit metric (0 or 1 per expected doc). The new `Recall@K` is the proper IR recall: retrieved_relevant / total_relevant, where relevant = all chunks for expected docs. This is the standard retrieval metric and is distinct from `doc_recall`.

**MRR@K:** straightforward — first rank where a relevant chunk appears. Captures whether the BEST result is relevant, regardless of recall depth.

**DB import fix:** `_fetch_collection_chunk_counts` initially used `p4_db.get_connection()` (dev/indexing DB module connecting to `rag_test`). Changed to `src.rag.db` (`rag` database via .env) — the `rag_test` database is the dev indexing test database and is empty in the current environment. The production collection data lives in `rag`.

**`_check_servers` fix:** the existing function unconditionally checked the embedding server at port 8081 even for BM25/sparse modes that don't use dense embeddings. Fixed to skip the embedding check for modes in `PREFIX_NOOP_MODES` (sparse, bm25). This is a correctness fix, not a workaround.

## Why Eval Execution Was Deferred

The task prompt explicitly deferred full eval runs: a parallel worker (`persisted-output-probe`) is calibrating the `top_k` threshold for the persisted-output mechanism. Retrieval top-k is directly tied to output size — running a sweep before top-k is fixed would produce results against a configuration that may change immediately after. The current `top_k=20` in BASELINE is the assumed production value but is pending confirmation.

Smoke test scope was limited to verifying metric field presence and mathematical correctness (6 unit test cases), not measuring real retrieval quality. The `RAG_MCP_test` collection was cleaned up in the April 30 data clean-slate; a full eval run requires re-indexing the test collection or running against a substitute.

## Why March 2026 NDCG Was Removed from Decisions

Three reasons:

1. **Not reproducible:** the code that produced the March tables is not in the repo. Any re-run of those experiments would require reconstructing the eval setup from scratch, making the numbers unverifiable from the current codebase.

2. **Not appropriate as IST evidence:** decision files' Evidenz section should reflect what the CURRENT pipeline has actually measured. March data was produced by a different eval stack, against different collections (SearXNG 26088 chunks, qwen3_paper 66 chunks), with a different eval harness. The current production pipeline runs against RAG_MCP_test (483 chunks) and will be evaluated by `A_retrieval_eval.py`.

3. **Decisions already made:** the March data drove two decisions — CC α=0.8 over RRF (retrieval03), and rerank=False as default (retrieval04). Both are codified in the Recommendation sections. The evidence is no longer load-bearing for any pending decision.

The March data is preserved verbatim in `OldThemes/eval_suite/historical_setup_march2026.md` for historical attribution.
