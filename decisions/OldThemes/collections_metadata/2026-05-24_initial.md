# Collections Metadata Table — 2026-05-24 Initial Design

## Why This Table Is Needed

Eval sweeps over different indexing configs (chunk size, embedding model, with/without sparse) require per-collection provenance: which model was used, what chunk size, which sparse model (or none). Without a machine-readable record, this information is only in report MDs and decision docs — grep-only, not queryable.

Three concrete triggers:

1. **Sweep attribution:** chunk-size sweep will produce `test_db`, `test_db_2`, `test_db_3` (2000/1000/512 chars). Eval reports need to JOIN against "what was the chunk_size for this collection?" without reading the index report MD.
2. **Self-describing system:** `rag-cli list_collections` currently returns only name + chunk count. With this table, the CLI can surface model + chunk_size inline — no context-switch to decisions docs needed.
3. **SPLADE on/off experiment gate:** if the SPLADE removal decision proceeds (evidence-gated from the mode sweep), new collections will be indexed without sparse embeddings. `sparse_model = NULL` is the machine-readable flag that distinguishes them from legacy collections that have `naver/splade-v3`.

## Schema

```sql
CREATE TABLE IF NOT EXISTS collections (
    name TEXT PRIMARY KEY,
    embedding_model TEXT,
    embedding_dims INTEGER,
    sparse_model TEXT,        -- NULL when indexed without sparse
    chunk_size INTEGER,
    overlap INTEGER,
    db_name TEXT,             -- 'rag' or 'rag_test'
    indexed_at TIMESTAMPTZ,
    doc_count INTEGER,
    chunk_count INTEGER,
    notes TEXT
)
```

Per-column rationale:

| Column | Rationale |
|---|---|
| `name` | Natural PK — collection names are unique within a DB, but two DBs can share the same name (hence `db_name` column) |
| `embedding_model` | Human-readable model name (e.g. `Qwen3-Embedding-8B`) — server URL is deployment detail, not provenance |
| `embedding_dims` | Full stored dimension (4096 for current Qwen3 config). MRL truncation is retrieval-time; storing retrieval dim would mislead future readers about what's in the DB |
| `sparse_model` | Nullable. `NULL` = dense-only collection. Current value: `naver/splade-v3` |
| `chunk_size` | Chars (not tokens) — matches the chunker's native unit |
| `overlap` | Chars overlap at chunk boundaries |
| `db_name` | `'rag_test'` or `'rag'`. Needed because the table lives in the same DB it describes; if we ever cross-query or dump, the DB context is otherwise lost |
| `indexed_at` | `TIMESTAMPTZ` not `TIMESTAMP` — psycopg2 passes TZ-aware datetimes; plain TIMESTAMP would silently drop TZ info |
| `doc_count` | Document count at index time. For full-collection re-index, equals number of source MDs processed |
| `chunk_count` | Total chunks inserted. The primary scale indicator for eval cost estimation |
| `notes` | Free-text. Used for: backfill annotation, partial-index flags, out-of-band config notes |

No secondary index beyond PK — the table will have at most ~20 rows (one per collection variant), full scan is negligible.

## Scope: rag_test Only (This Round)

Production `rag` DB stays untouched. Reasons:

1. **Risk isolation:** schema migration on production DB requires coordinating with `src/rag/indexer.py` (the production indexer) and `src/rag/db.py` (production connection layer). That is a separate worker task with its own review gate.
2. **Test-first:** rag_test validates the upsert behavior (full re-index overwrites, schema is idempotent) before the pattern propagates to production.
3. **No production indexing in current sprint:** the chunk-size sweep and SPLADE mode sweep only touch rag_test. Production metadata backfill is not blocking any current eval work.

Production migration plan: add `ensure_collections_schema()` equivalent to `src/rag/db.py`, call it from `src/rag/indexer.py::index_json_workflow()` alongside the existing `ensure_schema()`, add `upsert_collection_metadata()` call at the end of `index_json_workflow()` after successful index, then backfill all existing `rag` DB collections from known configs in the decisions MDs.

## Indexer Integration Points

**Dev pipeline (current implementation):**

- Schema creation: `ensure_collections_schema(conn)` in `dev/indexing/p4_db.py`, called from `dev/indexing/A_index_collection.py::run_index()` alongside `ensure_schema()`.
- Upsert: `upsert_collection_metadata(conn, ...)` in `dev/indexing/p4_db.py`, called from `run_index()` after `_indexer.index_directory()` returns successfully, before `conn.close()`.
- Trigger: every full index run. `A_index_collection.py` always clears the collection before indexing (`clear_collection()`), so there is no partial re-index path in this script — the upsert always reflects the full collection.
- Behavior on re-index: `ON CONFLICT (name) DO UPDATE SET ...` overwrites all fields. This is correct: a full re-index with different params should replace the previous metadata row entirely.

**Model/config constants** added to `A_index_collection.py` INFRASTRUCTURE:
- `EMBEDDING_MODEL = "Qwen3-Embedding-8B"`
- `EMBEDDING_DIMS = 4096`
- `SPARSE_MODEL = "naver/splade-v3"`
- `DB_NAME = "rag_test"`

These are passed verbatim to `upsert_collection_metadata`. Stats (`doc_count`, `chunk_count`) come from the `stats` dict returned by `index_directory()`.

## Open Questions

### Partial Re-index Behavior

**Decision (implemented):** upsert on FULL re-index only. `A_index_collection.py` always calls `clear_collection()` before indexing, so there is no partial re-index path in the dev script — the question is moot here.

For the production indexer (`src/rag/indexer.py`), partial re-index IS possible: `index_json_workflow()` deletes per-document before re-inserting, so adding a single new document leaves existing chunks intact. In that case, the correct upsert behavior is:

- **NOT** use the batch stats (would record doc_count=1, chunk_count=N for the new doc only).
- **DO** query aggregate counts from the DB after the partial index: `SELECT COUNT(DISTINCT document), COUNT(*) FROM documents WHERE collection = %s`.

This is the implementation pattern for the production migration. The decision: upsert on full re-index (A_index_collection.py pattern), query-then-upsert on partial re-index (production pattern). Capture in the production migration worker brief.

### Collections Renamed via MCP Auto-Collection Routing

The MCP Auto-Collection feature (scoping in `decisions/OldThemes/mcp_auto_collection/2026-05-24_scoping.md`) may introduce automatic collection name assignment. If a collection is renamed post-indexing (e.g. from a temp name to a canonical routing name), the `collections` row needs to follow:

- Option A: `UPDATE collections SET name = 'new_name' WHERE name = 'old_name'` — simple rename, preserves all metadata.
- Option B: re-index under the new name, let the upsert create the new row, delete the old row manually.

Option A is safer (no re-embedding cost). Option B is correct if the rename coincides with a config change. Decision deferred to the MCP Auto-Collection worker — that feature will define the rename lifecycle.

### Production Migration Plan

When extending to the `rag` DB:

1. Add `ensure_collections_schema(conn)` to `src/rag/db.py` FUNCTIONS (same pattern as existing `ensure_schema`).
2. Call it from `src/rag/indexer.py::index_json_workflow()` alongside `ensure_schema()`.
3. Add `upsert_collection_metadata(conn, ...)` call at the end of `index_json_workflow()`, using query-aggregate pattern for doc/chunk counts (not batch stats, because production indexer is per-document).
4. Backfill: one backfill script for all existing `rag` DB collections. Known configs are in `decisions/index01_chunking.md`, `index02_dense_embedding.md`, `index03_sparse_embedding.md`. `indexed_at` for pre-existing collections will use NOW() with a note (no index reports available for the production DB).
5. Update `src/rag/db.py::query_collections()` to optionally JOIN against `collections` for enriched output in `rag-cli list_collections`.
