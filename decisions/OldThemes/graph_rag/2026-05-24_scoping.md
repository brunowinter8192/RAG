# Graph RAG — Scoping (2026-05-24)

## Pain

Projekte werden in dieser Pipeline rein textuell wahrgenommen — über Dateinamen, decisions-MDs, DOCS.md-Tabellen, Bead-Beschreibungen. Es gibt keinen visuellen Knotenpunkt aus dem hervorgeht wie Module zusammenhängen, welche Decisions welche Code-Bereiche betreffen, welche OldThemes-Topics auf welche src/-Files referenzieren, welche Sources auf welche Pipeline-Steps einwirken. Für orientierungsschwache Momente in größeren Projekten ist das ein echter Friction-Punkt — User sagt: "ich seh das Projekt nie visuell vor mir".

## Zwei mögliche Bedeutungen von "Graph RAG"

Vor jeder weiteren Diskussion muss unterschieden werden welche Richtung gemeint ist:

**Richtung A: Projekt-Visualisierung als Graph.** Code-Symbol-Graph (Funktionen, Klassen, Module mit Import-Kanten), Decisions-Graph (welche `decisions/<step>.md` referenziert welche Source-Files, welche `OldThemes/`-Topics führten zu welcher SOLL), Source-Inventory-Graph (welche Sources werden in welchen Decisions zitiert). Tool-mäßig: extrahieren via AST/ripgrep, rendern als interaktiver Graph (Pyvis / D3 / Obsidian-style Backlinks / Mermaid). Output: navigable Karte des Projekts. Retrieval ist hier vielleicht zusätzlich, aber primär ist Visualisierung.

**Richtung B: Graph-augmented Retrieval.** Klassisches GraphRAG-Paradigma (Microsoft GraphRAG, LightRAG, etc.): aus der Korpus werden Entities + Relations extrahiert und als Knowledge Graph gespeichert. Bei Retrieval wird neben dem chunk-basierten Search auch der KG durchsucht — Entity-Match führt zu verbundenen Entities, Multi-Hop-Reasoning wird möglich. Output: bessere Antworten auf Fragen die mehrere Quellen kombinieren müssen ("welche Decisions hängen mit Topic X zusammen UND welche dieser Decisions referenzieren Source Y"). Visualisierung ist hier möglich aber nicht das Ziel.

Beide sind kombinierbar — Richtung A löst den Orientierungs-Pain direkt, Richtung B würde ad-hoc unsere Suchen verbessern, ist aber implementatorisch deutlich größer.

## Erste Indikation

User-Pain ist "ich sehe das Projekt nie visuell" — das spricht klar für Richtung A als ersten Hebel. Richtung B könnte später draufkommen.

## Mechanik Richtung A (Skizze)

- **Quellen extrahieren:** Python-AST für src/ Symbol-Graph (Funktionen, Klassen, Imports), Markdown-Parser für decisions/ + DOCS.md (extrahiert Referenzen auf File-Pfade, OldThemes-Verlinkungen, Source-Inventory-Pointer), git log für Commit→File-Mapping
- **Knoten-Typen:** Module, Funktion, Decision-File, OldThemes-Topic, Source (extern), Bead
- **Kanten-Typen:** imports, calls, references, supersedes, references-source, decision-touches-module, oldtheme-led-to-decision
- **Speicher:** Graph als JSON (oder Neo4j wenn wir sowieso eine DB wollen, oder NetworkX in-memory für Read-only-Snapshots)
- **Rendering:** statisch (Mermaid für Doku-Embedding) ODER interaktiv (Pyvis HTML, einmalig rendered, im Browser navigierbar)

Skeptisch zu prüfen: lohnt der Tool-Bau-Aufwand gegenüber einem einfachen Markdown-Index der von Hand gepflegt wird? Bei wachsenden Projekten ja, bei kleinen wahrscheinlich Overkill — Bauchgefühl ist bei drei Modulen mit je 10 Funktionen reicht eine Liste; bei 30 Modulen mit Decisions-Kreuzreferenzen wird der Graph kritisch.

## Was wäre ein Probe-Schritt

`dev/graph_rag/probe_richtung_a.py`: ein Skript das gegen `RAG/` läuft, AST + Markdown extrahiert, einen Pyvis-HTML rendert. Output anschauen, beurteilen ob das den Pain wirklich adressiert. Wenn ja → SOLL-Diskussion über Hosting (lokal HTML, Live-Server, Obsidian-Vault als Anzeigefläche). Wenn nein → andere Richtung versuchen oder den Bead schließen.
